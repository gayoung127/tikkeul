# 오더북 도메인 모델

> 이 문서는 거래소의 핵심인 오더북(호가창)을 설명합니다.
> 비즈니스 정책(수수료율, 한도 등)은 [비즈니스 규칙](business-rules.md)을 참고하세요.
> 오더북의 기술적 구현 설계는 [오더북 설계](../architecture/orderbook-design.md)에서 다룹니다.

## 1. 오더북이란?

오더북(Order Book)은 **아직 체결되지 않은 주문들이 모여 있는 자료구조**다.

DB 테이블이 아니라 **메모리에 존재**한다. DB의 Order 테이블에도 주문이 저장되지만, 실시간 매칭은 메모리의 오더북에서 수행한다.

오더북은 두 개의 면으로 구성된다:

```
         오더북 (KRW-BTC 마켓)

  asks (매도 주문)              bids (매수 주문)
  ─ 낮은 가격이 위 ─            ─ 높은 가격이 위 ─

  52,000,000원 × 0.5 BTC       
  51,000,000원 × 1.0 BTC       ← best ask (최우선 매도호가)
  ──────────────── 스프레드 ────────────────
                                50,000,000원 × 0.8 BTC  ← best bid (최우선 매수호가)
                                49,000,000원 × 1.2 BTC
                                48,000,000원 × 0.3 BTC
```

| 용어 | 설명 |
|------|------|
| **asks** (매도 측) | 팔겠다는 주문 모음. 낮은 가격이 우선 |
| **bids** (매수 측) | 사겠다는 주문 모음. 높은 가격이 우선 |
| **best ask** | asks 중 가장 낮은 가격 (매수자가 가장 싸게 살 수 있는 가격) |
| **best bid** | bids 중 가장 높은 가격 (매도자가 가장 비싸게 팔 수 있는 가격) |
| **스프레드** | best ask - best bid (이 간격이 0이면 즉시 체결 가능) |
| **가격 레벨** | 같은 가격의 주문들이 모인 단위. 시간순으로 정렬 |

### ask/bid와 buy/sell의 관계

두 쌍의 용어는 서로 다른 맥락에서 사용되며 1:1로 매핑된다.

| 오더북 용어 | 주문 용어 | 의미 |
|:-----------:|:---------:|------|
| **bids** (매수 측) | **BUY** 주문 | "이 가격에 사겠다" |
| **asks** (매도 측) | **SELL** 주문 | "이 가격에 팔겠다" |

- **buy/sell**: Order의 속성. 사용자의 의도를 나타냄 (`order.side = BUY / SELL`)
- **bid/ask**: 오더북의 구조. 주문이 어느 쪽에 배치되는지를 나타냄

## 2. 오더북에 등록되는 주문, 안 되는 주문

| 주문 유형 | 오더북 등록 | 이유 |
|-----------|:-----------:|------|
| 지정가 (LIMIT) | ✅ | 가격을 지정하고 대기. 상대방이 올 때까지 오더북에 남음 |
| 시장가 (MARKET) | ❌ | 즉시 체결 시도 후 잔여는 취소. 대기하지 않음 |

따라서 **오더북에 남아있는 주문은 항상 지정가 주문**이다.
시장가 × 시장가 체결은 불가능하다. 시장가 주문은 오더북에 등록되지 않으므로 서로 만날 수 없다.

### 시장가 주문이 오더북에 등록될 수 없는 근본적 이유

오더북은 가격순으로 정렬된 자료구조다. 시장가 주문은 **가격(price)이 없다.** 가격이 없는 주문은 오더북의 어느 위치에도 배치할 수 없다. 이것은 구조적 제약이므로 변하지 않는다

## 3. 매칭 알고리즘

### 3.1 새 주문이 들어왔을 때

```
[새 주문 도착]
    │
    ▼
[주문 유형 확인]
    │
    ├─ LIMIT(지정가)
    │     │
    │     ▼
    │   [상대편 오더북에서 체결 가능한 주문 탐색]
    │     │
    │     ├─ 체결 가능 → 체결 처리 (반복)
    │     │     │
    │     │     └─ 잔여 있으면 → 오더북에 등록 (대기)
    │     │
    │     └─ 체결 불가 → 오더북에 등록 (대기)
    │
    └─ MARKET(시장가)
          │
          ▼
        [상대편 오더북에서 체결 가능한 주문 탐색]
          │
          ├─ 체결 가능 → 체결 처리 (반복)
          │     │
          │     └─ 잔여 있으면 → 즉시 반환 + 취소 (IOC)
          │
          └─ 체결 불가 → 즉시 반환 + 취소 (IOC)
```

### 3.2 체결 가능 조건

| Taker 주문 | 상대편(Maker) | 체결 조건 |
|-----------|--------------|-----------|
| 매수 (BUY) | asks (매도 측) | taker.price >= maker.price (지정가) 또는 무조건 (시장가) |
| 매도 (SELL) | bids (매수 측) | taker.price <= maker.price (지정가) 또는 무조건 (시장가) |

### 3.3 체결 수량 계산

```
체결 수량 = min(taker의 잔여 수량, maker의 잔여 수량)
체결 가격 = maker의 price (항상)
체결 금액 = 체결 가격 × 체결 수량
```

시장가 매수는 수량이 아닌 금액(quote_amount) 기반이므로:

```
체결 수량 = min(taker의 잔여 금액 / maker.price, maker의 잔여 수량)
```

## 4. 오더북과 DB의 관계

```
┌──────────────────────────────┐     ┌──────────────────────────────┐
│       오더북 (메모리)          │     │        DB (영구 저장)         │
│                              │     │                              │
│  asks: 매도 주문 (가격순)      │ ──▶ │  Order 테이블: 모든 주문 기록  │
│  bids: 매수 주문 (가격순)      │     │  Trade 테이블: 모든 체결 기록  │
│                              │     │  Balance: 잔액 상태            │
│  역할: 실시간 매칭             │     │  BalanceHistory: 변동 이력     │
│  특징: 빠름, 휘발성            │     │  역할: 영구 저장, 감사 추적     │
└──────────────────────────────┘     └──────────────────────────────┘
```

- 주문이 오더북에 등록될 때 DB에도 저장된다 (status=OPEN)
- 체결이 발생하면 DB에 Trade가 생성되고, Order/Balance가 업데이트된다
- 서버 재시작 시 DB에서 OPEN/PARTIALLY_FILLED 주문을 읽어 오더북을 복구한다

## 5. 케이스별 전체 추적 예시

모든 예시의 공통 전제:

```
마켓: KRW-BTC
수수료율: 0.05% (0.0005)
수수료 통화: 받는 자산에서 차감 (매수자→BTC, 매도자→KRW)
```

---

### Case 1: 지정가 매도 → 지정가 매수 (전량 체결)

> B가 "BTC 0.01개를 5천만원에 팔겠다"고 먼저 등록하고,
> A가 "BTC 0.01개를 5천만원에 사겠다"로 체결이 발생하는 기본 시나리오.

#### 초기 상태

```
매수자 A:  KRW  available=1,000,000  locked=0
매도자 B:  BTC  available=0.1        locked=0
오더북: 비어있음
```

#### Step 1: B가 지정가 매도 주문

```
요청: LIMIT-SELL, price=50,000,000, quantity=0.01
```

**B의 잔액 변동:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| BTC | 0.1 | **0.09** | 0 | **0.01** |

**Order 레코드 (B의 주문):**

| 필드 | 값 |
|------|-----|
| side | SELL |
| type | LIMIT |
| status | **OPEN** |
| price | 50,000,000 |
| quantity | 0.01 |
| remaining_quantity | 0.01 |
| filled_quantity | 0 |
| filled_amount | 0 |
| locked_amount | 0.01 |
| fee | 0 |

**BalanceHistory (1건):**

| type | coin | amount | available_before | available_after | locked_before | locked_after | idempotency_key |
|------|------|--------|:-:|:-:|:-:|:-:|-----|
| LOCK | BTC | 0.01 | 0.1 | 0.09 | 0 | 0.01 | ORDER:{B주문ID}:LOCK:{BTC_ID} |

**오더북:**

```
asks: [50,000,000원 × 0.01 BTC (B의 주문)]
bids: (비어있음)
```

#### Step 2: A가 지정가 매수 주문

```
요청: LIMIT-BUY, price=50,000,000, quantity=0.01
```

**A의 잔액 락:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| KRW | 1,000,000 | **500,000** | 0 | **500,000** |

락 금액 = 50,000,000 × 0.01 = 500,000원

**BalanceHistory (1건):**

| type | coin | amount | available_before | available_after | locked_before | locked_after |
|------|------|--------|:-:|:-:|:-:|:-:|
| LOCK | KRW | 500,000 | 1,000,000 | 500,000 | 0 | 500,000 |

#### Step 3: 매칭 발생

A의 매수 주문(taker)이 오더북의 asks를 탐색:
- best ask = B의 주문 (50,000,000원)
- A의 price(50,000,000) >= B의 price(50,000,000) → **체결 가능**

```
체결 가격   = 50,000,000 (maker인 B의 price)
체결 수량   = min(A의 0.01, B의 0.01) = 0.01 BTC
체결 금액   = 50,000,000 × 0.01 = 500,000원
매수자 수수료 = 0.01 × 0.0005 = 0.000005 BTC (받는 BTC에서 차감)
매도자 수수료 = 500,000 × 0.0005 = 250원 (받는 KRW에서 차감)
```

**Trade 레코드:**

| 필드 | 값 |
|------|-----|
| buy_order_id | A의 주문 |
| sell_order_id | B의 주문 |
| maker_order_id | **B의 주문** (먼저 오더북에 있었음) |
| taker_order_id | **A의 주문** (체결을 발생시킴) |
| price | 50,000,000 |
| quantity | 0.01 |
| amount | 500,000 |
| buyer_fee | 0.000005 |
| seller_fee | 250 |

#### Step 4: 정산

**매수자 A:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| KRW | 500,000 | 500,000 (변동 없음) | 500,000 | **0** |
| BTC | 0 | **0.009995** | 0 | 0 |

- KRW locked -= 500,000 (체결금액 차감)
- BTC available += 0.01 - 0.000005 = 0.009995 (체결수량 - 수수료)

**매도자 B:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| BTC | 0.09 | 0.09 (변동 없음) | 0.01 | **0** |
| KRW | 0 | **499,750** | 0 | 0 |

- BTC locked -= 0.01 (체결수량 차감)
- KRW available += 500,000 - 250 = 499,750 (체결금액 - 수수료)

**BalanceHistory (6건):**

| # | type | 대상 | coin | amount | idempotency_key |
|---|------|------|------|--------|-----|
| 1 | BUY_SETTLE | A | KRW | 500,000 | TRADE:{ID}:BUY_SETTLE:{KRW_ID} |
| 2 | BUY | A | BTC | 0.009995 | TRADE:{ID}:BUY:{BTC_ID} |
| 3 | FEE | A | BTC | 0.000005 | TRADE:{ID}:FEE:{BTC_ID} |
| 4 | SELL_SETTLE | B | BTC | 0.01 | TRADE:{ID}:SELL_SETTLE:{BTC_ID} |
| 5 | SELL | B | KRW | 499,750 | TRADE:{ID}:SELL:{KRW_ID} |
| 6 | FEE | B | KRW | 250 | TRADE:{ID}:FEE:{KRW_ID} |

**Order 최종 상태:**

| | A의 주문 (매수) | B의 주문 (매도) |
|---|---|---|
| status | **FILLED** | **FILLED** |
| filled_quantity | 0.01 | 0.01 |
| remaining_quantity | 0 | 0 |
| filled_amount | 500,000 | 500,000 |
| fee | 0.000005 | 250 |
| filled_at | 체결 시각 | 체결 시각 |

**오더북:** 비어있음 (양쪽 모두 전량 체결)

### Case 2: 지정가 매도 → 시장가 매수 (전량 체결 + 잔여금 반환)

> B가 "BTC 0.01개를 5천만원에 팔겠다"고 등록하고,
> A가 "KRW 60만원어치 BTC를 지금 당장 사겠다"로 시장가 매수.
> 50만원에 체결되고 남은 10만원은 반환된다.

#### 초기 상태

```
매수자 A:  KRW  available=1,000,000  locked=0
매도자 B:  BTC  available=0.1        locked=0
오더북: 비어있음
```

#### Step 1: B가 지정가 매도 주문

Case 1과 동일. B의 BTC 0.01이 락 → 오더북 asks에 등록.

```
오더북 asks: [50,000,000원 × 0.01 BTC (B의 주문)]
```

#### Step 2: A가 시장가 매수 주문

```
요청: MARKET-BUY, quote_amount=600,000
```

**A의 잔액 락:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| KRW | 1,000,000 | **400,000** | 0 | **600,000** |

시장가 매수는 quote_amount 전체를 락한다.

**A의 Order 레코드:**

| 필드 | 값 |
|------|-----|
| side | BUY |
| type | MARKET |
| time_in_force | **IOC** |
| status | OPEN |
| price | **NULL** (시장가) |
| quantity | **NULL** (시장가 매수) |
| quote_amount | **600,000** |
| remaining_quote | 600,000 |

#### Step 3: 매칭 발생

```
체결 가격   = 50,000,000 (maker B의 price)
체결 수량   = min(A의 잔여금액 / maker.price, B의 잔여수량)
            = min(600,000 / 50,000,000, 0.01)
            = min(0.012, 0.01)
            = 0.01 BTC (B의 물량이 먼저 소진)
체결 금액   = 50,000,000 × 0.01 = 500,000원
매수자 수수료 = 0.01 × 0.0005 = 0.000005 BTC
매도자 수수료 = 500,000 × 0.0005 = 250원
```

**Trade 레코드:** Case 1과 동일한 구조 (maker=B, taker=A)

#### Step 4: 정산

Case 1과 동일. A는 BTC 0.009995, B는 KRW 499,750 수령.

#### Step 5: 잔여금 반환 (IOC 처리) ← Case 1과 다른 핵심 포인트

A의 quote_amount 600,000 중 500,000이 체결되었고, 오더북에 더 이상 매도 주문이 없다.
시장가(IOC)이므로 잔여 100,000원을 즉시 반환한다.

**A의 잔여금 반환:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| KRW | 400,000 | **500,000** | 100,000 | **0** |

**BalanceHistory 추가 (1건):**

| type | 대상 | coin | amount | idempotency_key |
|------|------|------|--------|-----|
| REFUND | A | KRW | 100,000 | ORDER:{A주문ID}:REFUND:{KRW_ID} |

**A의 Order 최종 상태:**

| 필드 | 값 |
|------|-----|
| status | **CANCELLED** (IOC — 잔여 취소) |
| quote_amount | 600,000 |
| remaining_quote | 0 |
| filled_quantity | 0.01 |
| filled_amount | 500,000 |
| cancelled_at | 취소 시각 |

> **핵심 차이**: 시장가 주문은 체결 후 잔여가 있으면 오더북에 등록하지 않고 즉시 반환 + CANCELLED 처리한다.
> 시장가는 가격이 없으므로 오더북에 등록하는 것 자체가 불가능하다.

> **CANCELLED 상태의 구분**: 시장가 부분 체결 후 CANCELLED와 사용자 취소 CANCELLED는 동일한 상태값이다.
> `filled_quantity > 0`이면 체결이 있었던 것, `filled_quantity == 0`이면 체결 없이 취소된 것으로 구분한다.
>
> | 상황 | status | filled_quantity | filled_amount |
> |------|--------|:-:|:-:|
> | 시장가 부분 체결 후 잔여 취소 | CANCELLED | > 0 | > 0 |
> | 시장가 미체결 (유동성 없음) | CANCELLED | 0 | 0 |
> | 사용자가 직접 취소 | CANCELLED | 0 또는 > 0 | 0 또는 > 0 |

### Case 3: 지정가 매수 → 지정가 매도 (부분 체결)

> A가 "BTC 0.05개를 5천만원에 사겠다"로 먼저 대기하고,
> B가 "BTC 0.02개를 5천만원에 팔겠다"로 부분 체결.
> A의 주문 중 0.03은 오더북에 남는다.

#### 초기 상태

```
매수자 A:  KRW  available=5,000,000  locked=0
매도자 B:  BTC  available=0.1        locked=0
오더북: 비어있음
```

#### Step 1: A가 지정가 매수 주문

```
요청: LIMIT-BUY, price=50,000,000, quantity=0.05
락 금액 = 50,000,000 × 0.05 = 2,500,000원
```

**A의 잔액 변동:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| KRW | 5,000,000 | **2,500,000** | 0 | **2,500,000** |

오더북에 상대 매도 주문이 없으므로 → 그대로 bids에 등록.

```
오더북 bids: [50,000,000원 × 0.05 BTC (A의 주문)]
```

#### Step 2: B가 지정가 매도 주문

```
요청: LIMIT-SELL, price=50,000,000, quantity=0.02
```

**B의 잔액 락:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| BTC | 0.1 | **0.08** | 0 | **0.02** |

오더북 bids를 탐색: A의 주문(50,000,000) 발견. B의 price(50,000,000) <= A의 price(50,000,000) → **체결 가능**

#### Step 3: 매칭 발생

```
체결 가격   = 50,000,000 (maker인 A의 price)
체결 수량   = min(B의 0.02, A의 0.05) = 0.02 BTC ← A의 물량이 남음
체결 금액   = 50,000,000 × 0.02 = 1,000,000원
매수자 수수료 = 0.02 × 0.0005 = 0.00001 BTC
매도자 수수료 = 1,000,000 × 0.0005 = 500원
```

**Trade 레코드:**

| 필드 | 값 |
|------|-----|
| maker_order_id | **A의 주문** (먼저 오더북에 있었음) |
| taker_order_id | **B의 주문** (체결을 발생시킴) |
| price | 50,000,000 |
| quantity | 0.02 |
| amount | 1,000,000 |

#### Step 4: 정산

**매수자 A:**

| 자산 | 변동 내용 |
|------|-----------|
| KRW | locked: 2,500,000 → **1,500,000** (-1,000,000) |
| BTC | available: 0 → **0.01999** (+0.02 - 0.00001) |

**매도자 B:**

| 자산 | 변동 내용 |
|------|-----------|
| BTC | locked: 0.02 → **0** (-0.02) |
| KRW | available: 0 → **999,500** (+1,000,000 - 500) |

#### Step 5: 주문 상태 업데이트

| | A의 주문 (매수) | B의 주문 (매도) |
|---|---|---|
| status | **PARTIALLY_FILLED** | **FILLED** |
| filled_quantity | 0.02 | 0.02 |
| remaining_quantity | **0.03** | 0 |
| filled_amount | 1,000,000 | 1,000,000 |

**오더북:**

```
bids: [50,000,000원 × 0.03 BTC (A의 주문, 잔여)] ← 아직 남아있음
asks: (비어있음)
```

> **핵심 포인트**: A의 주문은 PARTIALLY_FILLED 상태로 오더북에 0.03 BTC가 남아있다. 다음 매도 주문이 들어오면 다시 매칭된다.

### Case 4: 여러 가격 레벨 → 시장가 매도 (다단계 체결)

> 오더북에 서로 다른 가격의 매수 주문 3건이 대기 중이고,
> C가 시장가 매도로 여러 가격에 걸쳐 체결되는 시나리오.

#### 초기 상태

오더북 bids에 3건의 매수 주문이 등록되어 있다고 가정:

```
오더북:
  asks: (비어있음)
  bids:
    51,000,000원 × 0.01 BTC (A의 주문) ← best bid (가장 높은 가격)
    50,000,000원 × 0.02 BTC (B의 주문)
    49,000,000원 × 0.03 BTC (D의 주문) ← 가장 낮은 가격

매도자 C:  BTC  available=0.05  locked=0
```

#### Step 1: C가 시장가 매도 주문

```
요청: MARKET-SELL, quantity=0.025
```

**C의 잔액 락:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| BTC | 0.05 | **0.025** | 0 | **0.025** |

#### Step 2: 다단계 매칭

시장가 매도는 bids에서 **가장 높은 가격부터** 순서대로 체결한다.

**1차 체결: C ↔ A (best bid)**

```
체결 가격 = 51,000,000 (maker A의 price)
체결 수량 = min(C의 잔여 0.025, A의 0.01) = 0.01
체결 금액 = 51,000,000 × 0.01 = 510,000
C의 잔여  = 0.025 - 0.01 = 0.015 ← 아직 남음
```

| Trade #1 | 값 |
|----------|-----|
| maker | A |
| taker | C |
| price | 51,000,000 |
| quantity | 0.01 |
| amount | 510,000 |
| buyer_fee (A) | 0.000005 BTC |
| seller_fee (C) | 255원 |

**2차 체결: C ↔ B (다음 가격 레벨)**

```
체결 가격 = 50,000,000 (maker B의 price)
체결 수량 = min(C의 잔여 0.015, B의 0.02) = 0.015
체결 금액 = 50,000,000 × 0.015 = 750,000
C의 잔여  = 0.015 - 0.015 = 0 ← 전량 소진
```

| Trade #2 | 값 |
|----------|-----|
| maker | B |
| taker | C |
| price | 50,000,000 |
| quantity | 0.015 |
| amount | 750,000 |
| buyer_fee (B) | 0.0000075 BTC |
| seller_fee (C) | 375원 |

D의 주문에는 도달하지 않음 (C의 물량이 먼저 소진).

#### Step 3: 최종 상태

**매도자 C의 총 수령액:**

```
Trade #1: 510,000 - 255 = 509,745원
Trade #2: 750,000 - 375 = 749,625원
총 수령: 1,259,370원 (평균 체결 단가 ≈ 50,374,800원/BTC)
```

**Order 상태:**

| | A (매수) | B (매수) | D (매수) | C (매도) |
|---|---|---|---|---|
| status | FILLED | PARTIALLY_FILLED | OPEN | **FILLED** |
| filled_qty | 0.01 | 0.015 | 0 | 0.025 |
| remaining | 0 | **0.005** | 0.03 | 0 |

**오더북:**

```
bids:
  50,000,000원 × 0.005 BTC (B의 주문, 잔여)
  49,000,000원 × 0.03 BTC  (D의 주문, 변동 없음)
```

> **핵심 포인트**: 시장가 주문은 여러 가격 레벨을 소화할 수 있다. 가격이 불리해질 수 있으므로 이를 "슬리피지(Slippage)"라고 한다. C는 51,000,000원과 50,000,000원 두 가격에 걸쳐 체결되었다.

### Case 5: 시장가 매수 → 오더북 비어있음 (유동성 없음)

> 오더북에 매도 주문이 하나도 없는 상태에서 시장가 매수를 시도하는 시나리오.

#### 초기 상태

```
매수자 A:  KRW  available=1,000,000  locked=0
오더북: 완전히 비어있음
```

#### Step 1: A가 시장가 매수 주문

```
요청: MARKET-BUY, quote_amount=500,000
```

**A의 잔액 락:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| KRW | 1,000,000 | **500,000** | 0 | **500,000** |

#### Step 2: 매칭 시도

오더북 asks가 비어있음 → **체결 불가**

#### Step 3: IOC 처리 — 전액 반환

체결된 것이 없으므로 락된 전체 금액을 즉시 반환한다.

**A의 잔액 반환:**

| 자산 | available (전) | available (후) | locked (전) | locked (후) |
|------|:-:|:-:|:-:|:-:|
| KRW | 500,000 | **1,000,000** | 500,000 | **0** |

**BalanceHistory (2건):**

| # | type | coin | amount | idempotency_key |
|---|------|------|--------|-----|
| 1 | LOCK | KRW | 500,000 | ORDER:{ID}:LOCK:{KRW_ID} |
| 2 | REFUND | KRW | 500,000 | ORDER:{ID}:REFUND:{KRW_ID} |

**A의 Order 최종 상태:**

| 필드 | 값 |
|------|-----|
| status | **CANCELLED** |
| quote_amount | 500,000 |
| filled_quantity | 0 |
| filled_amount | 0 |

> **핵심 포인트**: 시장가 주문은 오더북에 유동성이 없으면 체결이 불가능하다. IOC 정책에 의해 잔여는 즉시 반환되고 주문은 CANCELLED 처리된다.

## 6. 서버 재시작 시 오더북 복구

오더북은 메모리에 존재하므로 서버가 재시작되면 사라진다.
복구 절차:

```
[서버 시작]
    │
    ▼
[DB에서 OPEN, PARTIALLY_FILLED 상태 주문 조회]
    │   SELECT * FROM `order`
    │   WHERE status IN ('OPEN', 'PARTIALLY_FILLED')
    │   ORDER BY created_at ASC
    │
    ▼
[오더북에 주문 재등록]
    │   - 매칭은 수행하지 않음 (기존 상태 그대로 복원)
    │   - remaining_quantity 기준으로 등록
    │
    ▼
[오더북 복구 완료 → 새 주문 수신 시작]
```

> 복구 시 매칭을 수행하지 않는 이유: 서버가 내려간 사이에 가격 조건이 바뀌었을 수 있고,
> 의도하지 않은 체결을 방지하기 위함이다. 이 정책은 Phase 2에서 재검토할 수 있다.
